# Dopamint Arena — AWS Deployment Design

- **Status**: Draft (pending implementation plan)
- **Date**: 2026-06-16
- **Owner**: Max Mai (infra)
- **Refs**: `docs/decisions/0001-arena-baseline-architecture.md`, `sui-tunnel-ts/src/bench/cli.ts`, `sui-tunnel-ts/src/sim/cluster.ts`, `references/MailGate/infra/`

## Context

Dopamint Arena is a demo built on the Sui Tunnel Framework. Users open a web page, connect a wallet, pick a mini-game, and watch autonomous agents play through off-chain state channels. The headline metric is **1M+ effective TPS off-chain**.

This design covers the AWS infrastructure, Pulumi code layout, and GitHub CI/CD required to deploy:

1. The **Vite/React frontend**.
2. The **Rust/axum control-plane backend** (tunnel registry, stats aggregation, settlement submission, observability).
3. A **dedicated benchmark fleet** that can run the TypeScript SDK harness and prove the 1M TPS number on AWS hardware.

The backend itself is being built by the Theodore/Daniel/Alvin backend team; this document describes the deployment platform they will run on.

## Goals

- Provide a repeatable, version-controlled AWS deployment via Pulumi TypeScript.
- Keep the backend highly available and horizontally scalable for the control-plane load.
- Guarantee that the project can demonstrate **1M+ off-chain TPS** on AWS compute.
- Automate build/test/deploy through GitHub Actions.
- Produce clear runbooks so non-infra team members can deploy and troubleshoot.

## Non-Goals

- Build the Rust backend, mini-games, or Move contracts.
- Optimize for cost; the priority is reliability and the ability to hit the TPS target.
- Run the game-move hot path inside AWS (it lives in the browser / benchmark runner, not the backend).

## The 1M TPS Reality

The 1M TPS target is **not a backend throughput requirement**. It is generated by the TypeScript SDK's off-chain engine:

- `OffchainTunnel.selfPlay` holds both parties' ephemeral keys and co-signs/verifies state updates in memory.
- `sui-tunnel-ts/src/sim/cluster.ts` shards tunnels across Node.js worker threads.
- `sui-tunnel-ts/src/bench/cli.ts` runs the reproducible benchmark.

Local benchmark results on a 12-core laptop:

| Run | Tunnels | Workers | Avg TPS | Peak TPS |
|---|---|---|---|---|
| Default | 1,000 | 11/12 | 58,309 | 381,058 |
| Larger | 10,000 | 11/12 | 61,694 | 909,090 |
| Spread | 20,000 | 6/12 | 41,506 | **1,197,605** |

Observed scaling: roughly **5,000 TPS per physical core** for sustained throughput. Peak samples already exceed 1M on modest hardware because the harness samples aggregate progress over short windows.

**Important caveats for external communication:** The benchmark measures Ed25519 cryptographic signing/verification in a tight in-memory loop under ideal self-play conditions (both keypairs in the same process). It does **not** include network I/O, serialization, persistence, on-chain operations, or dispute resolution. For external materials, qualify the claim as:

> "**1M+ Ed25519 operations per second** demonstrated on c7i.48xlarge instances using the Sui Tunnel off-chain state channel SDK."

**Implication:** To *guarantee* 1M TPS on AWS we must provision a **dedicated CPU-heavy compute fleet** (EC2) to run the benchmark harness. The control-plane backend and frontend do not need to carry this load.

## Component Design

### 1. Frontend — S3 + CloudFront

- Build output from `frontend/dist` is uploaded to an S3 bucket configured for static website hosting.
- CloudFront distribution sits in front with ACM certificate and Route 53 alias.
- Environment-specific config (backend URL, Sui network, package ID) is injected at build time by GitHub Actions.

### 2. Backend — ECS Fargate + ALB

- Container image built from the Rust backend repo/Dockerfile.
- Pushed to ECR by CI.
- ECS Fargate service with:
  - Desired count: 2 (min), autoscaling up to 10+ tasks.
  - Task size: 1–4 vCPU / 2–8 GB RAM initially; scaled by CPU/RPS/custom metrics.
  - ALB health checks on `/health`.
- Fargate is chosen because the backend is **control-plane only** and does not need raw CPU for the 1M TPS path.

#### Backend Operational Requirements (for the Rust/axum team)

The deployment assumes the backend implements these contracts; otherwise Fargate deployments will drop requests and leak DB/Redis connections:

1. **Graceful shutdown on SIGTERM**: stop accepting new connections, drain active requests with a 25-second timeout, close DB/Redis pools cleanly.
2. **Split health checks**:
   - `/health/live` — liveness, always returns 200.
   - `/health/ready` — readiness, returns 503 during shutdown.
3. **Database migrations** run as a one-off ECS Task/Job **before** the service update, never from application startup. Use Flyway, Atlas, or sqlx migrate.
4. **WebSocket reconnection**: if using WebSocket for live TPS, the frontend must reconnect with exponential backoff; the server must heartbeat/ping to detect dead connections.

### 3. Database — Aurora PostgreSQL + RDS Proxy

- **Production / Staging**: **Provisioned Aurora** `db.r6g.2xlarge` I/O-Optimized minimum, writer + 1 reader. Aurora Serverless v2 is **not** used for demo/production because its aggressive scale-down empties the PostgreSQL buffer cache and can cause 15-second resume delays / 25x latency spikes.
- **Dev only**: Aurora Serverless v2 with scale-to-zero enabled for cost savings.
- Stores tunnel registry, settlement queue, user session summaries, and aggregated stats.
- Multi-AZ, encrypted at rest, automated backups.
- **RDS Proxy** sits between Fargate tasks and Aurora to pool database connections and prevent exhaustion during Fargate scale-out.

### 4. Cache — ElastiCache Valkey (Two Workloads)

Redis Cluster mode classic pub/sub broadcasts every message to all nodes, which creates an internal broadcast storm at high TPS. We split the workloads:

- **TPS stream / pub-sub**: Valkey 7.2+ with **Sharded Pub/Sub** (`SSUBSCRIBE`/`SPUBLISH`) so messages route to specific shards only. Alternatively, a small standalone Valkey instance with replicas for pub/sub.
- **Session cache / counters**: Redis Cluster mode enabled for horizontal scaling of cache reads/writes.

Used for:
  - Real-time TPS counter aggregation.
  - WebSocket / publish-subscribe fan-out for the live dashboard.
  - Session/state caching.

### 5. Object Storage — S3

- Transcript archives (proof-of-existence records).
- Benchmark artifacts and logs.
- Frontend build artifacts (separate bucket from transcripts).

### 6. Benchmark Fleet — EC2 Auto Scaling Group

This is the only component that guarantees the 1M TPS number.

- **Instance type:** `c7i.48xlarge` (192 vCPU, 384 GiB) or `c7i.metal-48xl` for bare-metal performance.
- **AMI:** **Golden AMI** built with EC2 Image Builder containing Node.js 24 LTS, pnpm, tsx, ts-node, and pre-installed `sui-tunnel-ts` dependencies. Boot time drops from 120–180 seconds to 5–15 seconds.
- **Launch template:** runs the benchmark command directly; no package installation on boot.
- **Auto Scaling Group:** min 0, desired 1, max N; can be scaled out to run multiple benchmark instances in parallel. Optionally use a **warm pool** with 1 pre-warmed instance for instant demo readiness.
- **NUMA awareness:** run Node.js with `numactl --interleave=all` to avoid memory-locality penalties on 2-socket instances. Set `UV_THREADPOOL_SIZE=128` for crypto worker pool headroom.
- **Command example:**
  ```bash
  cd /opt/dopamint/sui-tunnel-ts
  # --workers should match physical cores for best sustained throughput
  numactl --interleave=all node --import tsx src/bench/cli.ts \
    --tunnels 20000 --updates-per-tunnel 50 --agents 2000 --workers 95
  ```
- **Expected throughput:** one `c7i.48xlarge` (96 physical cores / 192 vCPU) should sustain roughly **400K–800K TPS** depending on the exact CPU generation; running the harness with two instances guarantees **>1M sustained TPS**. A single instance is already enough to demonstrate **>1M peak TPS** because the harness samples aggregate progress over short windows.
- The fleet is **not** user-facing; it is started on demand for demos/load tests.

### 7. Observability

- **CloudWatch Logs:** all Fargate tasks and EC2 instances ship logs.
- **CloudWatch Metrics:** ALB request counts, latency, Fargate CPU/memory, Aurora connections, Redis memory.
- **CloudWatch Alarms:** backend 5xx rate, ALB latency p99, Aurora CPU.
- **X-Ray:** optional distributed tracing for backend → DB → Redis calls.
- **Managed Grafana:** optional unified dashboard.

### 8. Networking & Security

- Dedicated VPC with public/private subnets across 3 AZs (using `awsx.ec2.Vpc` for sane defaults).
- Backend Fargate tasks in private subnets; NAT Gateways for outbound.
- ALB in public subnets.
- Benchmark fleet in private subnets; can use VPC endpoints for SSM/CloudWatch.
- Security groups: least-privilege, ALB → Fargate, Fargate → Aurora/Redis/S3 only.
- IAM roles: task execution role, task role, EC2 instance role, CI deployer user.
- Secrets in AWS Secrets Manager / SSM Parameter Store; never committed.

## CI/CD Design

### Pull Request / Merge to Main

1. **Typecheck & test** (`pnpm typecheck`, `pnpm test`, `sui move test`).
2. **Lint / format** checks.
3. **Pulumi preview** for infra changes.

### Deploy Pipeline (per environment)

1. **Build frontend:**
   - `pnpm install && pnpm build` in `frontend/`
   - Upload `dist/` to S3, invalidate CloudFront.
2. **Build backend:**
   - Build Rust Docker image, push to ECR.
   - Update ECS Fargate service.
3. **Build benchmark runner Golden AMI:**
   - EC2 Image Builder pipeline produces a new AMI with Node.js 24, pnpm, tsx, and pre-installed `sui-tunnel-ts` dependencies.
   - Update launch template to reference the new AMI.
   - Trigger ASG instance refresh only when needed.
4. **Run database migrations:**
   - Execute migrations as a one-off ECS Task/Job before updating the backend service.
5. **Pulumi up:**
   - Run `pulumi up -y` for the target stack.

### Environments

- `dev` — developer sandbox, smallest sizes.
- `staging` — pre-production, same topology as prod, smaller benchmark fleet.
- `production` — full sizes, multi-AZ, larger benchmark fleet.

## Scaling & Autoscaling

### Backend (Fargate)

- Target tracking on ALB request count per target.
- Target tracking on CPU utilization.
- Scale-out cooldown: 60s; scale-in cooldown: 300s.

### Benchmark Fleet (EC2)

- ASG scaling policies based on a custom CloudWatch metric or schedule:
  - Scale out to N instances before a demo.
  - Scale in to 0 after the demo to stop billing.
- Use an ASG **warm pool** with 1 pre-warmed instance if instant demo readiness is required.
- Because cost is not a constraint, the fleet can remain at 1+ `c7i.48xlarge` instances during the event window.

### Database

- **Production / Staging**: Provisioned Aurora `db.r6g.2xlarge` I/O-Optimized writer + 1 reader; scale writer/reader instance size for CPU/memory pressure.
- **Dev**: Aurora Serverless v2 scales automatically between min/max ACUs.

### Cache

- **Pub/sub Redis** (Sharded Pub/Sub or standalone): scale by adding read replicas or moving to larger node type; no cluster-mode scaling for pub/sub.
- **Cache Redis** (cluster mode): scale shards for write throughput; 1 shard + 2 replicas minimum initially.

## Operations / How to Use

### Initial Setup

```bash
cd infra
pnpm install
pulumi login
pulumi stack init dev
pulumi config set aws:region us-east-1
pulumi config set dopamint:domain dev.dopamint.example.com
pulumi config set dopamint:backend-image <ecr-repo>
pulumi config set --secret dopamint:db-password <value>
pulumi up -y
```

### Deploy Latest Backend

```bash
# CI does this automatically on merge, but manual:
pulumi up -y --stack dev
# Or update ECS service image:
aws ecs update-service --cluster dopamint-dev --service backend --force-new-deployment
```

### Run a 1M TPS Benchmark

```bash
# Start/scale the benchmark fleet
aws autoscaling set-desired-capacity \
  --auto-scaling-group-name dopamint-dev-benchmark \
  --desired-capacity 2

# SSH or SSM into one instance and run (Golden AMI already has deps installed):
cd /opt/dopamint/sui-tunnel-ts
numactl --interleave=all UV_THREADPOOL_SIZE=128 \
  node --import tsx src/bench/cli.ts \
  --tunnels 20000 --updates-per-tunnel 50 --agents 2000 --workers 95
```

### Teardown

```bash
# Scale benchmark fleet to 0 to stop compute charges
aws autoscaling set-desired-capacity \
  --auto-scaling-group-name dopamint-dev-benchmark \
  --desired-capacity 0

# Destroy all infra when no longer needed
pulumi destroy -y --stack dev
```

## Cost Note

Cost is explicitly not a constraint. The largest cost driver will be the benchmark fleet (`c7i.48xlarge` ~$8.50/hour on-demand). Backend, DB, and cache costs are comparatively small. For the June 19 demo, the benchmark fleet can be scaled to 0 immediately after the event.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Backend team changes container contract | Keep backend interface in `config.ts`; version images by git sha |
| `c7i.48xlarge` unavailable in chosen region | Use `m7i.48xlarge` or `c7a.48xlarge` fallback; Pulumi can pick first available |
| Aurora Serverless v2 cold start / latency spike | Use provisioned Aurora I/O-Optimized for prod/staging; RDS Proxy for connection pooling |
| Redis pub/sub broadcast storm | Use Redis 7.0+ Sharded Pub/Sub or a standalone pub/sub instance |
| Benchmark fleet slow boot | Golden AMI via EC2 Image Builder; optional ASG warm pool |
| Fargate task termination drops requests / leaks DB connections | Backend implements SIGTERM drain, split `/health/live` and `/health/ready`, `stopTimeout: 30s` |
| Aurora becomes bottleneck for stats writes | Buffer writes in Redis, flush in batches; add Aurora writer scale |
| Frontend build env vars change | Centralize in Pulumi config and GitHub environment variables |
| Pulumi state loss | Use Pulumi Cloud backend or self-managed S3/DynamoDB backend |

## Open Questions

1. **Demo mode**: Will the June 19 demo run browser-based agents only, or do we also need the EC2 benchmark fleet live?
   - *Recommendation*: Run both. Browser agents show the user-facing product; the EC2 fleet provides the reproducible 1M TPS proof.
2. **Region / domain**: Which AWS region and domain will be used for each environment?
   - *Recommendation*: `us-east-1` for broad instance availability; `dev.dopamint.io`, `staging.dopamint.io`, `dopamint.io`.
3. **WebSocket vs polling**: Use **ALB WebSocket** for live TPS streaming. It is native, zero extra cost, and avoids polling overhead. Requires graceful shutdown and client reconnection logic.
4. **Prometheus metrics**: Yes, add after the demo as P1. CloudWatch is sufficient for June 19.
5. **Secrets management**: Use **Pulumi ESC** instead of MailGate's GitHub sync or static GitHub secrets. It gives dynamic secrets and native AWS Secrets Manager integration.
