# Design: ed25519 batch verification for off-chain TPS benchmark

**Date:** 2026-06-20  
**Branch:** `feat/offchain-tps-optimize`  
**Goal:** Increase honest full-sign TPS by replacing per-transition individual signature verification with a standard ed25519 batch-verification primitive, while preserving cryptographic correctness.

---

## Background

The off-chain TPS benchmark (`frontend/src/bench/offchainTps.ts`) measures transitions where both parties sign the canonical `state_update` and both signatures are verified. Current fleet results on 2× `c7i.48xlarge`:

| Mode | Fleet avg TPS |
|---|---|
| `none` | ~860k |
| `sign-only` | ~746k |
| `full` | ~410k |

`sign-only` shows that signing costs only ~13% of throughput, while adding verification drops throughput by ~45%. Batch verification amortizes the verify cost over many signatures and is the largest remaining optimization that does not relax the security model.

---

## Requirements

1. **Correctness:** A batch-verified signature set must be accepted only if every signature in the batch is individually valid under RFC 8032 / Sui's strict ed25519 rules.
2. **Fail-safe:** If a batch verify fails, the code falls back to individual verification to identify the invalid signature and abort the tunnel cleanly.
3. **Compatibility:** Existing cross-language tests (`crypto-native.test.ts`, `crypto.test.ts`, `signature_vectors_tests.move`, `wire_format_tests.move`) must continue to pass.
4. **Metric honesty:** The benchmark still counts every transition as fully dual-signed and dual-verified.
5. **No protocol changes:** The wire format, key format, and Move verifier remain unchanged.

---

## Architecture

### New `SignMode`: `"batch"`

Add a fourth mode alongside `full`, `sign-only`, and `none`:

- `full`: sign + individual verify per step (current behavior).
- `sign-only`: sign only, no verify.
- `none`: no crypto.
- `batch`: sign per step, defer verification until a batch of `N` signatures accumulates, then verify all at once.

### Backend API

Add `verifyBatch(messages: Uint8Array[], signatures: Uint8Array[], publicKeys: Uint8Array[]): boolean` to:

- `sui-tunnel-ts/src/core/crypto-native.ts` (Node/OpenSSL path)
- `sui-tunnel-ts/src/core/crypto.ts` (`@noble/curves` fallback path)

Both backends must:
- Reject non-canonical signatures and public keys.
- Use a CSPRNG-generated random scalar per batch (required by the batch-verify algorithm).
- Return `false` if any signature is invalid.

### Tunnel integration

`OffchainTunnel.step` in `sui-tunnel-ts/src/core/tunnel.ts` gains a small internal queue when `mode === "batch"`:

1. Sign the message as usual for both parties.
2. Append `(msg, sigA, pkA)` and `(msg, sigB, pkB)` to the queue.
3. When the queue reaches `batchSize` (configurable, default 64), call `verifyBatch`.
4. If batch verify returns `false`, run individual verification to find the bad signature and throw a verifiable error.
5. On tunnel close / terminal state, force-flush any remaining queued signatures.

### Benchmark integration

`frontend/src/bench/offchainTpsWorker.ts` accepts `--sign-mode batch` and reports `verifications` equal to the number of signatures verified (same total as `full`, just fewer `verifyBatch` calls).

---

## Algorithm

Use the standard ed25519 batch verification algorithm (Bernstein et al.):

- Compute `H(R_i || A_i || M_i)` for each signature.
- Generate random scalars `z_i` for i > 0.
- Check the single multi-scalar equation:
  `sum(z_i * s_i) * B == sum(z_i * R_i) + sum(z_i * h_i * A_i)`
- If the equation holds, all signatures are valid (with overwhelming probability).

The random scalars must be at least 128 bits and generated by a CSPRNG. We use `crypto.randomBytes` / `crypto.getRandomValues`.

---

## Testing

1. **Unit property test:** Generate 1,000 valid signatures; assert `verifyBatch` returns `true` and matches `every(verifyIndividual)`.
2. **Negative tests:**
   - Tamper with one message → batch fails.
   - Tamper with one signature → batch fails.
   - Use wrong public key for one entry → batch fails.
   - Inject a non-canonical signature → batch fails.
3. **Cross-backend parity:** Native backend and noble fallback must agree on all test vectors.
4. **Existing suite:** All Move and TS tests in `sui-tunnel-ts` and `frontend` must pass.

---

## Expected outcome

If batch verification is ~30–40% cheaper per signature than individual verification, and verify is ~45% of the full-mode cost, then `batch` mode should reach roughly **550–650k fleet TPS** on the same 2-instance setup.

---

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Non-canonical signature accepted by batch | Implement strict point/scalar checks; test against non-canonical vectors. |
| Bad RNG breaks batch security | Use CSPRNG; test that deterministic/fixed scalars are not used. |
| Async flush hides errors | Force-flush on terminal state; individual-verify fallback on batch failure. |
| Move compatibility break | Keep wire format and signature format unchanged; re-run Move cross-tests. |

---

## Scope

This change is limited to:
- `sui-tunnel-ts/src/core/crypto*.ts`
- `sui-tunnel-ts/src/core/tunnel.ts`
- `frontend/src/bench/offchainTpsWorker.ts`
- New tests in `sui-tunnel-ts/src/core/`

No frontend UI, Move contract, or protocol wire-format changes.
